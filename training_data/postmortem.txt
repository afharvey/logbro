Summary
An Azure Hyper-V host maintenance event (network VF reset) caused a 64GB managed disk to experience I/O failures and be reattached under a new device name. The Azure CSI driver did not detect or recover from the disk identity change, leaving a Neo4j database pod without a functional mount.

Root Cause
At 22:28:09 UTC on Jan 15, the Azure Hyper-V host performed a network Virtual Function (VF) switchover on node aks-dmv5skbo-94141891-vmss00000q. Kernel logs show:



hv_netvsc eth0: Data path switched from VF: enP64055s1
VF slot 1 removed
pci_bus fa37:00: busn_res: [bus 00] is released
This host-level maintenance event caused collateral I/O failures on the SCSI disk:



EXT4-fs warning (device sdg): ext4_end_bio:344: I/O error 10 writing to inode 524369
EXT4-fs (sdg): This should not happen!! Data will be lost
Aborting journal on device sdg-8.
Sequence of failure:

Disk was mounted as /dev/sdg at 22:26:45

Network VF reset occurred at 22:28:09–22:28:30

Immediate I/O errors on sdg at 22:28:30

EXT4 journal aborted at 22:28:36, filesystem corruption confirmed

Azure reattached the same disk as /dev/sdh at 22:34:55

CSI driver did not mount sdh — VolumeAttachment showed success, but no actual mount

Resolution
Deleted the affected pod, which triggered:

VolumeAttachment deletion

Clean disk detach

Pod reschedule with successful volume mount

Detection Gap
VolumeAttachment reported attached: true throughout

CSI driver logs showed no errors

Detection relied on Neo4j application-level Raft panic

13.5 hours elapsed before the issue was noticed

Lessons Learned
Azure host maintenance events can cause collateral disk failures

Disk device names (sdX) can change during Azure reattachment; CSI driver doesn’t handle this gracefully

Kubernetes VolumeAttachment status does not reflect node-level mount health

Node-level log analysis (kern.log) is essential for storage issues
