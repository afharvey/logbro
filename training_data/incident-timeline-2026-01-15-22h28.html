Executive Summary

At 22:28:09 UTC the Hyper-V host rescinded the SR-IOV virtual function (VF) backing the Mellanox/ConnectX NIC (enP64055s1), tearing down the entire PCIe bus fa37:00. Because Azure Premium Disk I/O is routed over the VMBus / hv_storvsc SCSI path and the in-flight write queue for /dev/sdg had already been submitted, the sudden bus removal caused every queued write to fail with hv 0xc0000044 (SRB_STATUS_ABORTED). This immediately produced cascading EXT4 I/O errors, a JBD2 journal abort, and confirmed data loss ("This should not happen!! Data will be lost") for Neo4j database write I/Os that were in flight at the time.

The disk was re-attached as /dev/sdh at 22:34:55, but the EXT4 journal on sdg remained aborted; sidecar and Neo4j processes continued hitting -EIO errors every 15 seconds through at least 22:39:55. A prior identical VF reset event was recorded at 14:39:10, indicating this is a recurring Hyper-V host-level event.

  • The root cause is a Hyper-V host-level SR-IOV VF rescind at 22:28:09 UTC. This is a platform event (live migration, host update, NIC firmware reset, or SR-IOV resource reclaim) and is not visible as a VM-level fault.
  • The VF tear-down removed the PCIe bus (fa37:00) shared by the Mellanox NIC. Although Azure Disk uses the separate VMBus/hv_storvsc path, the in-flight I/O completions were aborted when the VMBus channel reset during the 17-second re-establishment window (22:28:10 → 22:28:27).
  • The hv_storvsc error code hv 0xc0000044 = SRB_STATUS_ABORTED: the Hyper-V storage ring buffer drained all pending commands as aborted, not retried. EXT4 received these as write errors with error code 10 (mapped to -EIO at the BIO layer).
  • EXT4 delayed allocation (dax-off, ordered data mode) means that data pages were dirty in the page cache and had not yet been flushed. When the flush window coincided with the storvsc abort, those pages were lost permanently.
  • A prior identical event at 14:39:10 (904 suppressed storvsc callbacks, sdc write failures) confirms this is a recurring pattern on this node, not a one-off. The node survived that event without a journal abort only because the earlier failure was on sdc, which recovered in time.
  • Application affected: Neo4j (threads neo4j.DatabaseR, neo4j.RaftLogPr) and sidecar container — both running on the same PV mount of sdg.

Recommended Remediation Actions

  • Immediate: Unmount the aborted sdg EXT4 filesystem and run fsck.ext4 -f /dev/sdg (or sdh) from a clean boot to recover journal and repair inode table. Do not attempt to remount without fsck — the journal is aborted.
  • Immediate: Assess Neo4j data integrity — inodes 524314 and 524348 have confirmed write loss. Run Neo4j consistency check (neo4j-admin database check) before restarting the database.
  • Short-term: Enable errors=remount-ro on the EXT4 mount (already set on sda1, but not on sdg). This limits blast radius by remounting read-only on first error instead of continuing writes that will be lost.
  • Short-term: File an Azure support ticket for the node aks-dmv5skbo-94141891-vmss00000Q — two VF rescind events in one day (14:39 and 22:28) indicate a host-level hardware or firmware issue (NIC or hypervisor).
  • Medium-term: Consider using Azure Ultra Disk or Premium SSD v2 with disk.csi.azure.com/enable-async-attach: "false" and enable write barriers (-o barrier=1) to reduce the window of lost writes during transient storage interruptions.
  • Medium-term: For stateful workloads like Neo4j on AKS, configure Pod Disruption Budgets + PreStop hooks to gracefully flush writes before a node drain, and use Azure Disk ZRS (zone-redundant) storage to survive single-zone host events.
  • Long-term: Instrument hv_storvsc error rate as a Prometheus alert (node_disk_io_now + custom storvsc metrics) to detect callback suppression storms before they cascade to journal aborts.
